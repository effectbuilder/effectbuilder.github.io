RGBJunkie plugin developer guide

For third-party developers building USB HID lighting plugins that run inside the RGBJunkie desktop application. This document describes the public plugin ABI and the device API. It does not require access to RGBJunkie source code.

Downloads, licensing, templates, and tooling: rgbjunkie.com

Effects (Effect lighting mode) — HTML canvas effects are in the Effect developer guide (§12 worked HTML examples). Functional .mjs effects have a separate functional effects guide. Optional HTML copies for desktop testing live under examples/effects/. USB / device coverage — Supported devices (search VID/PID, transport, kind).

1. What a plugin is

  • A single JavaScript module (ES module syntax) loaded from RGBJunkie’s configured plugin directory (see app documentation for the path on your system).
  • The module exports export const rgbjunkie (ABI v1 descriptor) and initialize, render, and shutdown.
  • RGBJunkie runs the script in a sandbox and provides globals such as device (HID and layout), bus, and other stable entrypoints.

2. ABI descriptor (rgbjunkie)

Export an object with abiVersion: 1.

2.2 transportType

Use the exact spellings below for new plugins.

ValueWhen to use
HidDefault for USB Human Interface Device (LED / OUT reports). Preferred casing.
HIDSame as Hid; some plugins use all-caps—prefer Hid for new work.
SMBUSLighting over SMBus / I²C (e.g. DRAM, some GPUs).
networkNetwork-reachable controllers (e.g. virtual devices bundled with the app).
serialSerial or CDC-style binding when the host exposes that transport.

2.4 Settings row type

Each entry in rgbjunkie.settings must set type to one of:

typeUI role
booleanCheckbox
numberNumeric input
colorColor picker (often with min / max). String format: Section 2.5.
stringFree text
comboboxDropdown; use values array
listList-style control
selectDropdown; host may present like combobox
rangeNumeric range
sliderSlider

Also set default, and when needed min, max, step, values, optional valuesDisplay (same length as values: friendly combobox labels while stored values stay values[i]), group, description.

Optional visibleWhen toggles a row in the device settings panel: { property: "other_setting_id", equals: "Some mode" }, { property: "x", oneOf: ["a", "b"] }, or { property: "x", notEquals: false } (referenced by another setting’s id). For logical AND across multiple controls, use allOf: { allOf: [{ property: "mode", oneOf: ["A"] }, { property: "flag", equals: true }] } (when allOf is a non-empty array, top-level property / equals on the same object are ignored). The same object is honored on legacy getControllableParameters() rows as visibleWhen. Optional appendTokenToProperty on a combobox row: set it to another setting’s id (usually a multiline string field). Choosing a non-first option appends that token to the target value and resets the dropdown. Use a first values entry of "" as the idle choice; the host shows “Insert LHM tag…” for that row when appendTokenToProperty is set and there is no valuesDisplay[0], otherwise use valuesDisplay for friendly labels (same length as values).

2.5 Color syntax

RGBJunkie treats plugin-facing colors as 24-bit sRGB hex:

FormNotes
#RRGGBB Preferred. Leading # plus six hexadecimal digits (0–9, a–f; case-insensitive). Examples: #ff8800, #FFFFFF.
RRGGBB Six hex digits without # are accepted where documented (for example device.createColorArray prepends # internally).

Not accepted by the built-in hex parser: 3-digit CSS shorthand (#rgb), 8-digit #RRGGBBAA, spaces, or named colors (red, rgb(...)). Unparseable strings map to black (#000000) when converted to RGB—normalize in your plugin if you accept user text.

Settings (type: "color"): use default: "#rrggbb" (always six digits with #). The host color control expects this shape for initial display; other shapes may fall back until the user picks a color again.

2.6 Example

export const rgbjunkie = {
  abiVersion: 1,
  displayName: "Example RGB Strip",
  publisher: "Your Studio",
  match: { vendorId: 0x1234, productIds: [0x5678] },
  transportType: "Hid",
  deviceKind: "lightingcontroller",
  validateEndpoint: (endpoint) => endpoint.interface === 0,
  imageUrl: "", // e.g. https://assets.rgbjunkie.com/images/brands/nollie.png — see §2.8
  size: [18, 6],
  documentation: "example-strip",
  settings: [
    {
      id: "shutdownColor",
      group: "lighting",
      label: "Shutdown Color",
      type: "color",
      default: "#000000",
      min: "0",
      max: "360",
    },
  ],
};

2.7 Notes

  • match and matchAlternatives define every (VID, PID) pair that loads this plugin.
  • Each settings[].id becomes a named global in your plugin and persists with user profiles—keep IDs stable.
  • Optional ledChannels / resolveLedChannels(productId) can declare channels without using only device.addChannel in initialize.

2.8 Device tree thumbnails (imageUrl) and brand assets

In the app, Settings / device tree brand art follows src/ui/pluginDeviceImage.ts, using tiles at https://assets.rgbjunkie.com/images/brands/<slug>.png (lowercase slug; hyphens for spaces, e.g. thermal-take.png, nollie.png), in this order:

  1. rgbjunkie.imageUrl when it is already an https://assets.rgbjunkie.com/images/brands/… URL.
  2. Otherwise a slug from the plugin path; absolute Windows paths are normalized through /plugins/ so the first segment is never a drive letter (C: would wrongly become c).
  3. Folders _include, _templates, and Default are organizational (not brand names). All Nollie controllers are under plugins/Nollie/ (Nollie1, Nollie8 v2, Nollie16v3, Nollie32); Nollie*_Nos2.0.js and * CDC Nos2.1.js files there are alternate firmware/transport builds, not separate brands. Nollie*.js maps to nollie; DRGB*.js maps to drgb. The OemRgb/ folder groups shared OEM RGB HID plugins (_include/protocols/oem-rgb-hid.js); it is not one retail brand: DRGB*.js, JPU*.js, NEEDMAX*.js, and YICO*.js there map to drgb, jpu, needmax, yico instead of the folder name (same filename idea as scripts/update-plugin-brand-image-urls.mjs).
  4. Else publisher (canonicalized, e.g. RGBJunkie → rgbjunkie).

Set rgbjunkie.imageUrl to the brand tile URL when you ship one so the descriptor matches the tree.

  • Maintainers (RGBJunkie repo): npm run plugins:brand-image-urls runs scripts/update-plugin-brand-image-urls.mjs and sets quoted imageUrl inside export const rgbjunkie when a matching brand PNG exists in the inventory. Non-string imageUrl (e.g. another object’s field) is skipped.
  • Public device list: npm run docs:supported-devices rebuilds docs/supported-devices-data.json for supported devices; it re-reads each plugin file’s imageUrl so the page stays current without re-harvesting data/device-catalog.json. The Brand column matches the app (organizational folders, OemRgb/ OEM filename lines, then publisher).
  • Publish docs locally: npm run docs:sync-wamp copies guides and supported-devices assets into your web root (default C:\wamp64\www\api\docs).

3. Lifecycle exports

PhaseFunctionRole
Setup initialize() Open HID, register channels or layouts, register UI. May return a Promise.
Each frame render() Read host-filled colors, build packets, device.write / send_report. May return a Promise.
Shutdown shutdown(systemSuspending) Safe color when the app or OS shuts down.

RGBJunkie awaits Promises returned from these functions when you return them.

Example skeleton

export async function initialize() {
  device.addChannel("Main", 30);
  device.set_endpoint(0, 0x0001, 0xffc0, 0);
}

export async function render() {
  const ch = device.channel("Main");
  const rgb = ch.getColors("Inline", "RGB");
  const packet = buildVendorPacket(rgb);
  device.write(packet, packet.length);
}

export async function shutdown(systemSuspending) {
  const color = systemSuspending ? "#000000" : shutdownColor;
  device.write(buildSolidPacket(color), REPORT_LEN);
}

4. device API (summary)

Main methods for third-party lighting plugins. RGBJunkie focuses on lighting; non-RGB calls may be no-ops. Current capability statements: rgbjunkie.com.

4.1 HID output and endpoint selection

MethodRole
device.write(packet, length)Primary send path during render.
device.send_report(packet, length)Alternative path for output/feature reports.
device.set_endpoint(interface, usage, usage_page, collection?)Select HID collection before I/O; use with validateEndpoint.

For interrupt reads, prefer device.readAsync or device.input_reportAsync (equivalent behavior; naming varies by protocol style).

4.2 Channels and host-filled colors

MethodRole
device.addChannel / removeChannelNamed LED segments; colors come from the workspace/effect pipeline.
device.channel(name)LedCount(), getColors(format, order).
device.SetLedLimit(n)Cap total LEDs for discovery/UI.

4.3 Fixed-layout grids

MethodRole
device.setControllableLeds(ledNames[], positions[][])Register a per-cell LED grid.
export function getLedNames(), getLedPositions()Optional; RGBJunkie may call during startup.
device.color(x, y)Sample the fixed grid inside render.

4.4 Multiple zones (subdevices)

MethodRole
device.createSubdevice(name)Declare a zone.
device.setSubdeviceLeds(name, ledNames[], positions[][])LEDs and layout for the zone.
device.subdeviceColor(name, x, y)Sampled RGB for that zone in render.

4.5 Settings and parameters

MethodRole
device.addProperty(descriptor)Runtime sidebar/UI parameter.
device.setProperty / getPropertyRead/write registered parameters.

You may also export getControllableParameters(); RGBJunkie merges it with rgbjunkie.settings when both exist.

4.6 Displays and images

Some hardware exposes LCDs or image surfaces. Helpers such as setImageFromBase64, setSubdeviceImage, and getImageBuffer apply where supported—confirm for your RGBJunkie version on rgbjunkie.com.

5. Common plugin patterns

  1. Strip / hub — addChannel in initialize; channel(...).getColors in render; write with your packet format.
  2. Fixed LED grid — setControllableLeds or exported names/positions; device.color for mapping.
  3. Multiple zones — createSubdevice + setSubdeviceLeds; subdeviceColor per zone.
  4. Same protocol, many VID/PIDs — use matchAlternatives for OEM or regional IDs.

6. Shared code (@rgbj-include)

// @rgbj-include "relative/path/to/helpers.js"

Ship included files alongside your plugin. Only the main plugin file should export rgbjunkie and the lifecycle functions; includes are fragments.

7. Distribution and validation

Install your plugin (and includes) in the folder RGBJunkie documents for third-party plugins. Any bundled validation tools or templates are described with your download or on rgbjunkie.com.

RGBJunkie source tree: after large batches of plugin metadata changes, run npm run harvest:plugin-db so data/device-catalog.json stays aligned. For brand imageUrl sweeps and the public device browser, see §2.8.

8. Plugin add-ons (WLED)

In this section: 8.1 · 8.2 · 8.3 · 8.4 · 8.5 · 8.6 · 8.7

8.1 What an add-on is

For some network template plugins, RGBJunkie can merge an optional second JavaScript file after the main plugin body when virtual devices are created. Today this is used for WLED: the stock plugin is plugins/Network/WLED/WLED.js; the add-on extends it with extra settings and optional matrix/clock behavior without forking the whole driver.

The host concatenates (in order): a small prelude (globals such as controller, udp, XHR proxy), the main WLED.js body, then the add-on text, then a discovery bridge. Your add-on runs in the same sandbox eval unit as the base plugin, so names and lifecycle must not fight the stock exports.

8.2 File name and install location

  • Place the add-on next to the main file, same relative path under your plugin root (for example %APPDATA%\RGBJunkie\plugins\Network\WLED\ on Windows).
  • Use either wled.addon.js or WLED.addon.js (the host tries both).
  • The main plugin must remain WLED.js (or whatever path your install uses for the WLED driver); the add-on is never loaded as a standalone device row.

8.3 Building the WLED clock / matrix add-on from the repo

  1. Developers: you need a matrix-capable WLED.js source that contains the markers let WLED; and export function Initialize (the generator extracts the block between them). Run npm run gen:wled-clock-addon -- <path-to-that-WLED.js> or set WLED_MATRIX_PLUGIN_SRC to the same path and run node scripts/gen-wled-clock-addon.mjs. That writes dist/wled-clock-addon/wled.addon.sample.js (gitignored — not bundled with the app). Rename/copy it to wled.addon.js in your user plugin folder. The bundled settings support the full matrix surface (Components vs time, custom text with Small/Medium/Large/Chinese fonts, pixel art, Libre Hardware Monitor, scroll, RGBCW, shutdown soft-off, etc.) as implemented by your source file.
  2. End users: place wled.addon.js (and the glyph siblings WLED_Text.js / WLED_Text_ZH.min.js next to it, as referenced by @rgbj-include lines in the bundle) beside WLED.js under %APPDATA%\RGBJunkie\plugins\Network\WLED\ on Windows, then restart or rescan.
  3. The sample uses // @rgbj-include "./WLED_Text.js" and // @rgbj-include "./WLED_Text_ZH.min.js" for glyph tables. Those sibling files must exist next to the add-on in your install (or the include resolver will fall back to the same path under other plugin scan roots).
  4. Restart RGBJunkie or trigger a plugin rescan so configured WLED virtual devices pick up the new bundle.
  5. On desktop, enable Settings → Engine → Hot-reload user plugins to watch %APPDATA%\RGBJunkie\plugins for .js / .mjs saves and rebuild sandboxes without a full restart (same idea as effect HTML hot-reload). Only the user plugin folder is watched — not the bundled plugins/ tree inside the app install.

8.4 Rules so the base plugin keeps working

  • Only one “real” plugin descriptor. The merged unit must still expose a single export const rgbjunkie and the real initialize / render / shutdown from WLED.js.
  • Do not ship empty lifecycle stubs in included fragments. Glyph helper files that were written as standalone stub plugins may contain export function initialize() {} and export function render() {}. When those files are inlined, the Rust expander may rewrite export function to function; in a single new Function(…) eval, hoisted function declarations can cause the stub to override the real WLED lifecycle even if the stub text appears after the stock code in the concatenated source. Result: no Strip channel is registered and no UDP frames are sent. Remove those stubs from included libraries, or rely on a current RGBJunkie build that strips them when merging WLED add-ons.
  • Prefer includes for data and helpers only — constants, fonts, pure functions — not a second copy of rgbjunkie or lifecycle exports. Section 6 describes @rgbj-include; treat includes as fragments, not second entrypoints.
  • Extra settings can be merged into rgbjunkie.settings at load time (the sample uses a guarded push loop). Use unique id values so you do not collide with stock WLED settings keys.

8.5 Optional hook: __rgbjPostWirePluginExports

If the add-on exports export function __rgbjPostWirePluginExports(), the host may replace that function’s body with a built-in shim (same merge step as stub stripping) so glyph-helper pitfalls stay fixed. The shim wires initialize (runs __wledClockAfterInit after stock init), shutdown (matrix blackout + stock shutdown), and render: when __wledClockUseStockUdpCanvas() is true (default for Components / effect canvas), it calls stock render() (DRGB/DNRGB); when false (Time, Custom Text, Pixel Art, Libre Hardware Monitor), it calls WLED.SendColorPackets(false) so the matrix path runs.

To use your add-on file’s original __rgbjPostWirePluginExports verbatim (no host shim), set globalThis.__RGBJUNKIE_WLED_CLOCK_MATRIX_RENDER = true before WLED virtual plugins are injected.

8.6 Blueprint: custom WLED add-ons (read this first)

The matrix bundle produced by npm run gen:wled-clock-addon is long because it embeds a full matrix implementation between a small RGBJunkie header and footer. If you write a new add-on, keep yours short by following the same shape, not the same size.

  1. One eval unit. The host concatenates prelude + base WLED.js + your add-on + discovery bridge into a single module. Everything shares one global scope — avoid redeclaring const rgbjunkie or a second function initialize() {} that would override the stock lifecycle.
  2. Stock owns lifecycle. Base WLED.js registers the Strip channel and sends UDP in render(). Your add-on usually merges extra entries into rgbjunkie.settings (see §8.4) and optionally wraps exports.render from __rgbjPostWirePluginExports (§8.5).
  3. Add-on display name. Optional export const __rgbjWledAddonDisplayName = "…" in your add-on; RGBJunkie shows it as the heading above merged settings that use group: "matrix". If omitted, a generic WLED add-on label is used.
  4. @rgbj-include = fragments only. Include glyph tables or pure helpers. Do not include files that declare empty function initialize() {} / function render() {} — hoisting can replace the real WLED lifecycle (§8.4). The host strips common empty stubs when merging, but fixing includes is clearer.
  5. Matrix generator vs hand-written. The repo script only automates slicing a matrix WLED.js between let WLED; and export function Initialize, then injecting the RGBJunkie header/footer. A minimal custom add-on can skip the generator entirely: one wled.addon.js file with settings merge + your logic + __rgbjPostWirePluginExports if you need render interception.
  6. Source map in code. For the exact merge and shim behaviour, read (in order) createWledAuthoredPluginCode, stripEmptyLifecycleStubsFromMergedWledAddon, and rehostWledClockAddonForRgbJunkieUdpPath in src/core/system.ts, and the long comment at the top of scripts/gen-wled-clock-addon.mjs.

Minimal pattern (illustrative only):

/* Merge settings — unique ids; do not replace export const rgbjunkie */
(function () {
  try {
    var d = typeof rgbjunkie !== "undefined" ? rgbjunkie : null;
    if (!d || !Array.isArray(d.settings)) return;
    d.settings.push({
      id: "my_matrix_toggle",
      group: "matrix",
      label: "Example",
      type: "boolean",
      default: false
    });
  } catch (e) {}
})();

/* Optional: run after the host wires exports (see §8.5) */
export function __rgbjPostWirePluginExports() {
  try {
    if (!exports || typeof exports.render !== "function") return;
    var base = exports.render;
    exports.render = function () {
      if (typeof my_matrix_toggle !== "undefined" && my_matrix_toggle) {
        /* your UDP / layout path */
      }
      return base();
    };
    exports.Render = exports.render;
  } catch (e) {}
}

Today only WLED uses this sibling-file merge. Other network templates would need the same host support before they could load an add-on the same way.
