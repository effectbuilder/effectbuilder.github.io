RGBJunkie plugin developer guide

For third-party developers building USB HID lighting plugins that run inside the RGBJunkie desktop application. This document describes the public plugin API and the device API. It does not require access to RGBJunkie source code.

Downloads, licensing, templates, and tooling: rgbjunkie.com

Effects (Effect lighting mode) — HTML canvas effects are in the Effect developer guide (§12 worked HTML examples). Functional .mjs effects have a separate functional effects guide. Optional HTML copies for desktop testing live under examples/effects/. USB / device coverage — Supported devices (search VID/PID, transport, kind).

1. What a plugin is

  • A single JavaScript module (ES module syntax) loaded from RGBJunkie’s configured plugin directory (see app documentation for the path on your system).
  • The module exports export const rgbjunkie (v1 plugin descriptor with abiVersion: 1) and initialize, render, and shutdown.
  • RGBJunkie runs the script in a sandbox and provides globals such as device (HID and layout), bus, and other stable entrypoints.

2. Plugin descriptor (rgbjunkie)

Export an object with abiVersion: 1. That field is the version number the host checks when loading your module; the rest of this section documents the descriptor shape.

In this section: 2.1 · 2.2 · 2.3 · 2.4 · 2.5 Color syntax · 2.6 · 2.7 · 2.8

2.1 Required and common fields

FieldTypeMeaning
abiVersion1Version covered by this guide.
displayNamestringShown in the device list.
publisherstringYour name or organization (recommended).
match{ vendorId, productIds[] }Primary USB IDs for binding.
matchAlternativesarray of same shapeOptional. Extra VID/PID sets for the same protocol (combined with match).

2.2 transportType

Use the exact spellings below for new plugins.

ValueWhen to use
HidDefault for USB Human Interface Device (LED / OUT reports). Preferred casing.
HIDSame as Hid; some plugins use all-caps—prefer Hid for new work.
SMBUSLighting over SMBus / I²C (e.g. DRAM, some GPUs).
networkNetwork-reachable controllers (e.g. virtual devices bundled with the app).
serialSerial or CDC-style binding when the host exposes that transport.

2.4 Settings row type

Each entry in rgbjunkie.settings must set type to one of:

typeUI role
booleanCheckbox
numberNumeric input
colorColor picker (often with min / max). String format: Section 2.5.
stringFree text
comboboxDropdown; use values array
listList-style control
selectDropdown; host may present like combobox
rangeNumeric range
sliderSlider

Also set default, and when needed min, max, step, values, group, description.

2.5 Color syntax

RGBJunkie treats plugin-facing colors as 24-bit sRGB hex:

FormNotes
#RRGGBB Preferred. Leading # plus six hexadecimal digits (0–9, a–f; case-insensitive). Examples: #ff8800, #FFFFFF.
RRGGBB Six hex digits without # are accepted where documented (for example device.createColorArray prepends # internally).

Not accepted by the built-in hex parser: 3-digit CSS shorthand (#rgb), 8-digit #RRGGBBAA, spaces, or named colors (red, rgb(...)). Unparseable strings map to black (#000000) when converted to RGB—normalize in your plugin if you accept user text.

Settings (type: "color"): use default: "#rrggbb" (always six digits with #). The host color control expects this shape for initial display; other shapes may fall back until the user picks a color again.

2.6 Example

export const rgbjunkie = {
  abiVersion: 1,
  displayName: "Example RGB Strip",
  publisher: "Your Studio",
  match: { vendorId: 0x1234, productIds: [0x5678] },
  transportType: "Hid",
  deviceKind: "lightingcontroller",
  validateEndpoint: (endpoint) => endpoint.interface === 0,
  imageUrl: "", // e.g. https://assets.rgbjunkie.com/images/brands/nollie.png — see §2.8
  size: [18, 6],
  documentation: "example-strip",
  settings: [
    {
      id: "shutdownColor",
      group: "lighting",
      label: "Shutdown Color",
      type: "color",
      default: "#000000",
      min: "0",
      max: "360",
    },
  ],
};

2.7 Notes

  • match and matchAlternatives define every (VID, PID) pair that loads this plugin.
  • Each settings[].id becomes a named global in your plugin and persists with user profiles—keep IDs stable.
  • Optional ledChannels / resolveLedChannels(productId) can declare channels without using only device.addChannel in initialize.

2.8 Device tree thumbnails (imageUrl) and brand assets

In the app, Settings / device tree brand art follows src/ui/pluginDeviceImage.ts, using tiles at https://assets.rgbjunkie.com/images/brands/<slug>.png (lowercase slug; hyphens for spaces, e.g. thermal-take.png, nollie.png), in this order:

  1. rgbjunkie.imageUrl when it is already an https://assets.rgbjunkie.com/images/brands/… URL.
  2. Otherwise a slug from the plugin path; absolute Windows paths are normalized through /plugins/ so the first segment is never a drive letter (C: would wrongly become c).
  3. Folders _include, _templates, Default, and Firmware 2.x are organizational (not brand names); Nollie*.js maps to nollie; DRGB*.js maps to drgb. The OemRgb/ folder groups shared OEM RGB HID plugins (_include/protocols/oem-rgb-hid.js); it is not one retail brand: DRGB*.js, JPU*.js, NEEDMAX*.js, and YICO*.js there map to drgb, jpu, needmax, yico instead of the folder name (same filename idea as scripts/update-plugin-brand-image-urls.mjs).
  4. Else publisher (canonicalized, e.g. RGBJunkie → rgbjunkie).

Set rgbjunkie.imageUrl to the brand tile URL when you ship one so the descriptor matches the tree.

  • Maintainers (RGBJunkie repo): npm run plugins:brand-image-urls runs scripts/update-plugin-brand-image-urls.mjs and sets quoted imageUrl inside export const rgbjunkie when a matching brand PNG exists in the inventory. Non-string imageUrl (e.g. another object’s field) is skipped.
  • Public device list: npm run docs:supported-devices rebuilds docs/supported-devices-data.json for supported devices; it re-reads each plugin file’s imageUrl so the page stays current without re-harvesting data/device-catalog.json. The Brand column matches the app (organizational folders, OemRgb/ OEM filename lines, then publisher).
  • Publish docs locally: npm run docs:sync-wamp copies guides and supported-devices assets into your web root (default C:\wamp64\www\api\docs).

3. Lifecycle exports

PhaseFunctionRole
Setup initialize() Open HID, register channels or layouts, register UI. May return a Promise.
Each frame render() Read host-filled colors, build packets, device.write / send_report. May return a Promise.
Shutdown shutdown(systemSuspending) Safe color when the app or OS shuts down.

RGBJunkie awaits Promises returned from these functions when you return them.

Example skeleton

export async function initialize() {
  device.addChannel("Main", 30);
  device.set_endpoint(0, 0x0001, 0xffc0, 0);
}

export async function render() {
  const ch = device.channel("Main");
  const rgb = ch.getColors("Inline", "RGB");
  const packet = buildVendorPacket(rgb);
  device.write(packet, packet.length);
}

export async function shutdown(systemSuspending) {
  const color = systemSuspending ? "#000000" : shutdownColor;
  device.write(buildSolidPacket(color), REPORT_LEN);
}

4. device API (summary)

Main methods for third-party lighting plugins. RGBJunkie focuses on lighting; non-RGB calls may be no-ops. Current capability statements: rgbjunkie.com.

4.1 HID output and endpoint selection

MethodRole
device.write(packet, length)Primary send path during render.
device.send_report(packet, length)Alternative path for output/feature reports.
device.set_endpoint(interface, usage, usage_page, collection?)Select HID collection before I/O; use with validateEndpoint.

For interrupt reads, prefer device.readAsync or device.input_reportAsync (equivalent behavior; naming varies by protocol style).

4.2 Channels and host-filled colors

MethodRole
device.addChannel / removeChannelNamed LED segments; colors come from the workspace/effect pipeline.
device.channel(name)LedCount(), getColors(format, order).
device.SetLedLimit(n)Cap total LEDs for discovery/UI.

4.3 Fixed-layout grids

MethodRole
device.setControllableLeds(ledNames[], positions[][])Register a per-cell LED grid.
export function getLedNames(), getLedPositions()Optional; RGBJunkie may call during startup.
device.color(x, y)Sample the fixed grid inside render.

4.4 Multiple zones (subdevices)

MethodRole
device.createSubdevice(name)Declare a zone.
device.setSubdeviceLeds(name, ledNames[], positions[][])LEDs and layout for the zone.
device.subdeviceColor(name, x, y)Sampled RGB for that zone in render.

4.5 Settings and parameters

MethodRole
device.addProperty(descriptor)Runtime sidebar/UI parameter.
device.setProperty / getPropertyRead/write registered parameters.

You may also export getControllableParameters(); RGBJunkie merges it with rgbjunkie.settings when both exist.

4.6 Displays and images

Some hardware exposes LCDs or image surfaces. Helpers such as setImageFromBase64, setSubdeviceImage, and getImageBuffer apply where supported—confirm for your RGBJunkie version on rgbjunkie.com.

5. Common plugin patterns

  1. Strip / hub — addChannel in initialize; channel(...).getColors in render; write with your packet format.
  2. Fixed LED grid — setControllableLeds or exported names/positions; device.color for mapping.
  3. Multiple zones — createSubdevice + setSubdeviceLeds; subdeviceColor per zone.
  4. Same protocol, many VID/PIDs — use matchAlternatives for OEM or regional IDs.

6. Shared code (@rgbj-include)

// @rgbj-include "relative/path/to/helpers.js"

Ship included files alongside your plugin. Only the main plugin file should export rgbjunkie and the lifecycle functions; includes are fragments.

7. Distribution and validation

Install your plugin (and includes) in the folder RGBJunkie documents for third-party plugins. Any bundled validation tools or templates are described with your download or on rgbjunkie.com.

RGBJunkie source tree: after large batches of plugin metadata changes, run npm run harvest:plugin-db so data/device-catalog.json stays aligned. For brand imageUrl sweeps and the public device browser, see §2.8.
