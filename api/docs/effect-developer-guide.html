RGBJunkie effect developer guide

For anyone who wants to build HTML canvas effects that run when users choose Effect lighting mode in the RGBJunkie desktop application — including beginners who are still learning JavaScript. If that is you, read Simple motion & shapes first, then skim the worked examples below before diving into the numbered sections.

The rest of this page is a technical reference (metadata, sidebar controls, engine API). It does not require access to RGBJunkie source code.

Functional (.mjs) effects use one JavaScript module with sampleLed instead of a canvas. They have their own end-user guide (HTML) so this page stays focused on HTML canvas.

Effect Builder, gallery publishing, and downloads: rgbjunkie.com

Making simple effects (start here)

RGBJunkie effects are normal web pages with a <canvas>. You draw one frame at a time; the app shows that picture on your LEDs. You do not need to be an expert — many beautiful effects are just a handful of shapes moving in a pattern.

A simple animation loop

Browsers expose requestAnimationFrame: you register a function that runs before the next repaint, usually about 60 times per second. Inside that function you clear (or paint over) the canvas, draw your shapes using the current time, then schedule the next frame.

  • const ctx = canvas.getContext("2d") — the “pencil” for rectangles, circles, lines, gradients, and text.
  • canvas.width / canvas.height — pixel size RGBJunkie sets for you; always use these for layout (see §2.1).

Drawing basic shapes

Common building blocks on a 2D canvas:

  • Filled rectangle — ctx.fillStyle = "#3399ff"; ctx.fillRect(x, y, width, height);
  • Circle or disc — ctx.beginPath(); ctx.arc(cx, cy, radius, 0, Math.PI * 2); ctx.fill();
  • Line or trail — ctx.beginPath(); ctx.moveTo(x1, y1); ctx.lineTo(x2, y2); ctx.stroke();
  • Smooth color — build a string like "hsl(" + hue + ", 80%, 55%)" and assign it to ctx.fillStyle (hue from 0–360).

Why Math.sin and Math.cos are your friends

Math.sin and Math.cos always return numbers between −1 and 1. Feed them an angle that grows smoothly with time (for example performance.now() / 1000) and you get a gentle wave — perfect for motion and pulsing colors without jarring jumps.

  • Move in a circle — pick a center (cx, cy), a radius R, and an angle t in radians. Then x = cx + Math.cos(t) * R and y = cy + Math.sin(t) * R. Increase t each frame and a dot orbits the center.
  • Slide left and right — x = cx + Math.sin(t) * amplitude (no cos needed). The same idea works for vertical motion with y.
  • Breathe a color — map sine to 0…1: const u = Math.sin(t) * 0.5 + 0.5, then use u for hue, opacity, or the mix between two palettes.

Minimal pattern (drop inside your draw / frame loop after you have canvas and ctx):

const t = performance.now() / 1000;
const w = canvas.width, h = canvas.height;
const cx = w * 0.5, cy = h * 0.5;
const R = Math.min(w, h) * 0.28;
const x = cx + Math.cos(t) * R;
const y = cy + Math.sin(t) * R;

ctx.fillStyle = "#0a0a12";
ctx.fillRect(0, 0, w, h);
ctx.beginPath();
ctx.arc(x, y, 14, 0, Math.PI * 2);
ctx.fillStyle = "hsl(" + ((Math.sin(t * 0.6) * 0.5 + 0.5) * 360) + ", 85%, 58%)";
ctx.fill();

1. What an effect is

  • A single HTML document (markup, styles, scripts) loaded in an isolated iframe when the user selects your effect.
  • The host injects window.engine, window.points, RGBJunkie_Ready, and one global variable per setting from your <meta property> rows (Section 4).
  • You normally draw each frame to a <canvas>; the workspace engine samples it for LEDs and hardware.
  • Alternatively, a functional effect is a single .mjs file exporting meta and sampleLed — the host calls sampleLed per LED instead of sampling a canvas. See the functional effects guide.

2. Minimal document shape

Use HTML5 with metadata and settings in <head>. Make the page and canvas fill the iframe with CSS; the host sets the canvas bitmap size (Section 2.1).

<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8" />
  <title>My Effect</title>
  <meta name="developer" content="Your Name" />
  <meta name="description" content="Short description for the effect browser." />
  <meta name="tags" content="ambient, gradient" />

  <meta property="speed" label="Speed" type="number" min="1" max="100" default="40" />

  <style>
    html, body { margin: 0; height: 100%; overflow: hidden; background: #000; }
    canvas { display: block; width: 100%; height: 100%; }
  </style>
</head>
<body>
  <!-- Host sets canvas.width/height; omit fixed width/height unless you need a stub for testing outside RGBJunkie -->
  <canvas id="fx"></canvas>
  <script>
    const canvas = document.getElementById("fx");
    const ctx = canvas.getContext("2d");
    function draw() {
      const w = canvas.width, h = canvas.height;
      /* use w/h; do not assume fixed 320×200 */
    }
    function frame() { requestAnimationFrame(frame); draw(); }
    window.addEventListener("resize", function () { /* refresh cached layout if needed */ });
    requestAnimationFrame(frame);
  </script>
</body>
</html>

2.1 Canvas and workspace container

The effect iframe fills the workspace preview (width/height: 100%). Use html, body { height: 100%; } and stretch the <canvas> with width/height: 100% so the element always matches the panel.

Bitmap resolution (for 2D drawing) is set by the host on the largest <canvas> in the page: it matches the user’s Engine Resolution, but is never larger than the iframe’s client box — if the configured resolution would not fit, the bitmap is scaled down with aspect preserved. On change, the host dispatches resize on the effect window.

SourceMeaning
canvas.width, canvas.heightDrawable pixel size after host sync — use for coordinates.
engine.canvas.width, engine.canvas.heightLogical Engine Resolution. When the iframe is smaller than that size, the bitmap may be smaller — still draw using the element’s width/height.

Prefer a single primary canvas so the host targets the right surface.

3. Discovery metadata

The host reads the metas below when building the effect browser (search, filters, cards) and the compact effect details line in the sidebar. They are ordinary <meta> rows in <head> — not <meta property> settings.

RoleDeclaration
Developer<meta name="developer" content="…"> — also author, publisher, creator, or property equivalents.
Description<meta name="description" …> or property="description".
Tags<meta name="tags" content="a, b, c"> — comma-separated; keywords is an alias. Tags (with the name, developer, and description) feed search and style filters in the effect browser — use clear words users might look for (ambient, audio, spectrum, …) so your effect groups well with similar entries.

If omitted, the UI shows sensible placeholders.

Grouping and tooltips on <meta property> (sidebar)

The same <head> usually contains <meta property="…"> rows for user-facing controls (see section 4). Those rows are not part of the discovery parser above, but they do control how options appear in the workspace Effect panel:

  • Category tab (group) — Set group="Display", group="Audio", or any short tab name so the control lands in that top-level category. The alias setting-group (or settingGroup) is accepted. If you omit group, RGBJunkie picks a tab from the property name and label (e.g. properties whose names mention audio lean toward Audio).
  • Sub-tab from label — If the label contains :, the text before the first colon is a sub-tab inside the category, and the text after is the row title (example: label="Fans: Row 1 count"). Labels without : use the category’s default sub-tab (often General).
  • Tooltips — Add tooltip="Longer help text" on any <meta property> row. The host shows it as the row’s native hover tooltip (title) on the label and control strip so users can read a sentence or two without cluttering the label.

Full attribute reference: Section 4.1 (defaults, min/max, combobox values, …).

6. engine API (summary)

Use engine or window.engine. Capability statements may change by version — confirm on rgbjunkie.com.

MemberRole
engine.audiofreq — 200 bins (0…255); level, density. Example: Section 12.2.
engine.canvaswidth / height — Engine Resolution (logical). For drawing, use your <canvas> element’s width/height; they cap to the iframe when needed (Section 2.1).
engine.requestNativeScreenBitmap()Recommended. Returns Promise<ImageBitmap> — one frame of the user's desktop ready to draw. Zero arguments. Monitor, capture rate, and capture size are all host-managed: the app injects a Display picker, a Capture rate slider (FPS budget — the Promise resolves no faster than the chosen rate), and a Capture size slider (max-edge pixels) whenever an effect uses this API. Effects do not declare these as <meta property>.
engine.requestNativeScreenFrame()Raw-pixel variant for analysis effects (e.g. extracting a dominant color without drawing). Returns a Promise resolving to { width, height, data, toImageData() }. Same host-managed plumbing as the bitmap variant; legacy maxWidth/maxHeight/monitorIndex opts are accepted but ignored.
engine.listScreenMonitors()Promise of display rows for multi-monitor UI.
engine.systemScreen / engine.screenLast capture buffer (width, height, data, updatedAt).
engine.getScreenCaptureImageData()ImageData or null from the last native frame (same buffer as systemScreen).
engine.getSensorValue(name)Compatibility stub — verify before relying on it.

Screen capture: Full teaching listing in Section 12.3 (requestNativeScreenFrame, toImageData). The built-in Screen Ambience catalog effect is the production reference for an effect-side canvas-width control. Whenever an effect references either screen-capture API the app injects three sidebar controls: Display (which monitor to capture), Capture rate (FPS budget — the Promise resolves no faster than the chosen rate), and Capture size (max-edge pixels of the capture). Effects must not declare screen_monitor_index, screen_capture_fps, or screen_capture_size via <meta property>, must not pass maxWidth/maxHeight/monitorIndex, and must not hand-roll a setTimeout loop.

7. Rendering and compositing

  • Draw to a <canvas>; the host reads the largest canvas and maps pixels to LED output.
  • Fill the iframe with CSS; use canvas.width/height for drawing and listen for resize (Section 2.1).
  • Use requestAnimationFrame for animation loops.
  • Call requestNativeScreenFrame only when you need a frame; duplicate requests may share one OS capture.

8. Optional: keyboard → canvas coordinates (privacy)

  • The host sends only KeyboardEvent.code into the effect (physical key), never key / typed text.
  • Set window.__rgbjKeyboardSpotByCode to map each code to { x, y } in your canvas pixel space.
  • Implement onKeyboardCanvasPoint(x, y) to react to presses at those coordinates.
  • Nested combiner effects may set window.__rgbjForwardKeyEventsToChildLayers = true to forward the same payload to child iframes.

9. Logging

console.log, warn, error are forwarded to the host effect log so users can troubleshoot without iframe devtools.

12. Worked examples (full source)

Minimal teaching code. Each example below is the smallest file that exercises one API surface — no clamping, no fallbacks, no defensive coding. 12.1 Color profile gradient

How it works. Declaring <meta property="colorProfile" type="colorProfile"> tells the host to inject two globals into the effect: colorProfile (the profile id, useful for display) and colorProfileStops (a ready-to-draw closed array of hex strings — the host resolves Rainbow/None/Custom modes and closes the palette so the seam is invisible). The effect just reads colorProfileStops and paints. The actual teaching trick in this example is the seamless slide: the gradient is built twice the canvas width with both copies of the palette inside it, so as slide cycles 0 → 1 the visible region always falls within the gradient — no wrap visible.

<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8" />
  <title>Color profile gradient</title>
  <meta name="description" content="Animated gradient driven by the user's selected color profile." />

  <!--
    The `type="colorProfile"` meta tells RGBJunkie to inject TWO globals:
      * `colorProfile`      — the profile id (e.g. "Rainbow", "Sunset")
      * `colorProfileStops` — a ready-to-draw, closed hex-stop array
    Most effects only need `colorProfileStops`.
  -->
  <meta property="colorProfile" label="Color profile" type="colorProfile" default="Rainbow" />

  <style>
    html, body { margin: 0; height: 100%; background: #000; }
    canvas { display: block; width: 100%; height: 100%; }
  </style>
</head>
<body>
  <canvas id="fx"></canvas>
  <script>
    const canvas = document.getElementById("fx");
    const ctx = canvas.getContext("2d");

    function draw() {
      const stops = colorProfileStops;
      const w = canvas.width, h = canvas.height;
      const slide = (performance.now() / 5000) % 1;

      // The gradient is 2× wide and holds two copies of the palette, so the
      // visible canvas always falls inside it as `slide` cycles 0 → 1.
      const grad = ctx.createLinearGradient(-slide * w, 0, (2 - slide) * w, 0);
      for (let i = 0; i < stops.length; i++) {
        const p = (i / (stops.length - 1)) * 0.5;
        grad.addColorStop(p, stops[i]);
        grad.addColorStop(p + 0.5, stops[i]);
      }

      ctx.fillStyle = grad;
      ctx.fillRect(0, 0, w, h);
      requestAnimationFrame(draw);
    }
    draw();
  </script>
</body>
</html>

12.2 Audio spectrum bars

How it works. engine.audio.freq is a 200-element array of spectrum bins (each 0–255), refreshed in place by the native audio pipeline. The loop bins those 200 raw values down to BARS = 64 visualizer columns by sampling at even intervals — real audio visualizers always bin (most of the upper-frequency bins are near zero in normal music, so rendering all 200 looks blurry and busy). The <canvas> declares an explicit internal resolution of 640×320 so each bar lands on a whole-pixel boundary. Note the explicit black fillRect background each frame instead of clearRect — the host's canvas sampler treats fully transparent pixels (which clearRect produces) as "keep previous value", leaving a trail. Effects that paint a non-transparent background every frame (or fill the whole canvas with their gradient, like the previous example) sidestep this entirely.

<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8" />
  <title>Audio bars</title>
  <meta name="description" content="Vertical bars driven by the audio spectrum." />

  <style>
    html, body { margin: 0; height: 100%; background: #000; }
    canvas { display: block; width: 100%; height: 100%; }
  </style>
</head>
<body>
  <!-- Explicit internal resolution so each bar is a whole number of pixels wide. -->
  <canvas id="fx" width="640" height="320"></canvas>
  <script>
    const canvas = document.getElementById("fx");
    const ctx = canvas.getContext("2d");
    const BARS = 64;

    function draw() {
      // `engine.audio.freq` is a 200-element array of spectrum bins (each 0–255).
      // Bin down to BARS by sampling at even intervals — 64 bars is a comfortable
      // visualizer width, and most of the raw bins above the audible mid-range are
      // near zero anyway.
      const freq = engine.audio.freq;
      const w = canvas.width, h = canvas.height;
      const barW = w / BARS;

      // Paint an opaque black background each frame. `clearRect` would only zero the
      // alpha channel, and the host's per-frame canvas sampler treats fully
      // transparent pixels as "keep previous value" — that leaves a trail.
      ctx.fillStyle = "#000";
      ctx.fillRect(0, 0, w, h);
      for (let i = 0; i < BARS; i++) {
        const v = freq[Math.floor(i * freq.length / BARS)] / 255;
        ctx.fillStyle = `hsl(${(i / BARS) * 360}, 80%, 55%)`;
        ctx.fillRect(i * barW, h - v * h, barW, v * h);
      }
      requestAnimationFrame(draw);
    }
    draw();
  </script>
</body>
</html>

12.3 Screen sample

How it works. engine.requestNativeScreenBitmap() returns a Promise<ImageBitmap> — one frame of the user's desktop, ready to drawImage. The loop awaits it, paints it, closes the bitmap, then schedules the next iteration with requestAnimationFrame. Everything else is host-managed. Whenever an effect calls this API the app injects three sidebar controls automatically: Display (which monitor), Capture rate (FPS budget — the Promise does not resolve faster than the user-chosen rate), and Capture size (max-edge pixels of the capture). The effect itself takes no arguments, declares no screen-capture metas, and does not throttle its own loop. For a production-grade reference that also exposes an effect-side canvas-width control, see the shipped Screen Ambience effect.

<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8" />
  <title>Screen sample</title>
  <meta name="description" content="Draws a downscaled copy of the user's desktop into the canvas." />

  <style>
    html, body { margin: 0; height: 100%; background: #000; }
    canvas { display: block; width: 100%; height: 100%; }
  </style>
</head>
<body>
  <canvas id="fx"></canvas>
  <script>
    const canvas = document.getElementById("fx");
    const ctx = canvas.getContext("2d");

    async function loop() {
      // Monitor, capture rate, and capture size are all host-managed — the app injects
      // sidebar controls for them. The effect just asks for the next frame.
      const bmp = await engine.requestNativeScreenBitmap();
      ctx.drawImage(bmp, 0, 0, canvas.width, canvas.height);
      bmp.close();
      requestAnimationFrame(loop);
    }
    loop();
  </script>
</body>
</html>
